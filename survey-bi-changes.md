# survey-bi 修改计划

## Bug / 小问题修复

### 1. `renderRatioFields` 把 average 当 count 渲染

- **位置**：[js/survey-bi.js:854-860](js/survey-bi.js:854)
- **现状**：
  ```js
  function renderRatioFields(fields) {
    const rows = Object.entries(fields).map(([label, payload]) => ({
      label,
      count: Math.round(payload.average),
    }));
    return renderBars(rows, 100, rows.length);
  }
  ```
  把 0–100 的 `average`（比率题的平均百分比）塞进 `count` 字段，分母硬编码 100。结果 bar 长度对，但 `renderBars` 里会再次按 `count/max` 算柱宽，并显示 "N / x%" —— "N" 实际是百分数，跟其他柱图的"计数 / 占比"含义不一致，会误导读者。
- **修复方向**：为 ratio 题写专门的 `renderRatioBars`，标注单位为 `%`（不显示 count），或在 bar-value 字段上区分单位类型。同时考虑展示 min/max/分布，不只是均值。

### 2. `renderTextStats` 的 `public samples` 永远为 0

- **位置**：[js/survey-bi.js:862-871](js/survey-bi.js:862)
- **现状**：硬编码 `renderSimpleMetric('public samples', 0)`。清洗端 `TEXT_DETAIL_LIMIT = 0` 所以确实没样本，但前端写死会造成"以后开了样本输出忘改"的隐患。
- **修复方向**：改为 `result.samples?.length || 0`，让前端跟着数据走。同时如果将来开放样本，需要新增样本展示区（见下面"扩展建议 #11"）。

### 3. `aggregate_ranking` 的 score 跨年不可比

- **位置**：[scripts/clean_survey_data.py:734-769](scripts/clean_survey_data.py:734)
- **现状**：Borda 分公式 `max_rank + 1 - rank`，其中 `max_rank = len(question.options)`。2025 和 2026 因为成员毕业/加入，候选成员数不同 → score 的上限不同 → 直接横向比对会失真。看板上 [js/survey-bi.js:819](js/survey-bi.js:819) 的 ranking-table 没有任何标注。
- **修复方向**：
  - 输出"归一化 score"：`score / (max_rank * rankedCount)`，限定在 0–1 区间
  - 同时在 ranking-table 头部加 caveat：候选成员数 N=xx，score 区间不同年份不可直接比对
  - 或者只用 averageRank + top1/top3 做跨年对照，score 仅作单年内部使用

## 扩展建议

按"价值密度"分三档。

---

### 一、高价值（建议优先做）

#### A1. 加 segmentation / 筛选器 ★最大缺口

现在所有图表都是**总体均值**，无法回答：
- "2024 年后入坑的粉丝里，第一接触渠道是什么？"
- "女性 18–25 岁的 C 位预测是谁？"
- "没参与海外席的人，购票难点是什么？"

具体方案：
- 顶部一排 filter chip：`入坑期 / 性别 / 年龄 / 地区 / 日语水平 / 是否订阅 OFC / 是否参与海外席`
- 选中后所有 panel 重新过滤，并显示 "样本：N / 837"
- **难点**：当前 `aggregate-*.json` 已经按题聚合，没有行级数据。两种实现路径：
  - (a) 离线**预计算交叉表**：每对 segment × question 出一份聚合，文件膨胀但仍是静态 JSON
  - (b) 公开**脱敏行级 JSON**：把 `survey-private/cleaned-*-rows.json` 中的高基数文本字段 hash/丢弃后发出，前端实时算 → 灵活度高，需再过一遍隐私评估
- **倾向**：方案 (b)。220+837=1057 行 JSON 体积可控（gzip 后估计 <500KB），分析维度近乎无限。

#### A2. 让"年份对比"诚实承认 n=837 vs n=220

样本量差近 4 倍，但 compare-bars 视觉上完全等价。建议：
- 每个柱子加 **95% 置信区间须线**（Wilson 区间）
- panel 顶部加显著性标签："这 10 题里有 3 题是统计显著差异"（chi-square test，跑在 Python 端预算）
- 把"看上去变了但其实在置信区间内"的对比题用淡灰标注，避免读者过度解读

负责任的数据可视化基础动作，对一档"摒弃消费主义论述"的播客尤其重要。

#### A3. 粉丝旅程漏斗 (Engagement Funnel)

把分散题目串成链路：

```
入坑 → 浏览中文内容 → 订阅 OFC → 订阅 Message → 报名抽选 → 中签 → 现场参与 → 见面会/签名会 → 海外席
 100%   Q19 有平台     Q16 是    Q17 是          Q30 有难点  Q29 中过  Q22 有出现  Q26-Q28          Q29-Q30
```

每步显示留存率，**2025 vs 2026 双色叠加**。BI 经典图，能一眼看出"中文区粉丝在哪一步流失最多" —— 很可能就是从 OFC 订阅到参与抽选那一跳。

#### A4. 成员深度页 (Member detail)

点 ranking-table 任一成员 → 弹层 / 独立页：
- 历年好感排名变化（折线）
- top1 率 / top3 率 / 平均排名
- 哪些 cohort（按入坑期分）最喜欢 ta
- 这位成员的"进步奖得票 / C 位预测得票 / 见面会参与"全打通

成员是粉丝看板的核心实体，扁平表格远不够。

#### A5. 自由文本不再只统计长度

13 道文本题（2025: Q15/Q22/Q27/Q28/Q40 + 2026: Q14/Q17/Q38）目前只显示 `answeredTextCount / averageLength / maxLength`，是**严重的数据浪费**。清洗端加：
- 词频统计（jieba 分词 + 停用词，输出 top-50 词云数据）
- 成员名字典命中统计（"哪些成员被自然提及最多"）
- 简单情感极性（见面会感言、抽选体验题尤其有价值）

前端加"词云"或"top 关键词"面板。**所有输出仍是聚合统计，原文不出库**，符合现有隐私架构。

#### A6. 首推率 + 衍生指标族

排序题（2025-Q03, 2026-Q07）已经输出 `top1` 和 `rankedCount`，前端加一行除法即可。但**两种首推率含义完全不同**，必须区分展示：

| 指标 | 公式 | 测量的是 |
|---|---|---|
| **条件首推率** | top1 / rankedCount | 死忠浓度 — 认识 ta 的人里有多少把 ta 当最爱 |
| **绝对首推率** | top1 / sampleCount | 群众基础 — 全样本中有多少人把 ta 当最爱 |

**强制要求**：上线时**两种都必须显示**，且每个指标必须有 hover tooltip 解释（依赖 B7 通用 tooltip 组件）。原因：单看条件首推率，一个只被 30 人排过、其中 25 人推 ta 第一的小众成员会获得 83% 的可怕数字 —— 必须配合 `rankedCount` 一起看才不会误读。

##### 必做项

**(a) Top-N 推率族**：把首推率扩展成序列
- 首推率 = top1 / rankedCount
- 三推率 = top3 / rankedCount
- 入榜率 = rankedCount / sampleCount

三者组合后能划分四种成员类型：

| 入榜 | 首推 | 类型 |
|---|---|---|
| 高 | 高 | **天下第一推**（人人喜欢且很多当最爱）|
| 高 | 低 | **国民好感**（大家都喜欢但难当最爱）|
| 低 | 高 | **死忠真爱**（小众但粉狂热）|
| 低 | 低 | **认知度不足**（需要曝光）|

**(b) 首推占有率（"得票分布"）**：某成员 top1 / Σ 所有成员 top1。像选举得票率，全员加起来 = 100%。配合 100% stacked bar，直观看出"首推市场份额集中度"。

**(c) 入榜率 × 首推率 二维散点图**：横轴入榜率、纵轴条件首推率，每个成员一个点。这是 A6 的核心展示图，能一图展示上面的四象限。

##### 后续可加

**(d) 首推 - Borda score 错位指数**：某成员的首推排名 vs 综合排名差值。
- 首推第 1 / 综合第 5 → "少数核心的真爱"
- 首推第 8 / 综合第 2 → "大家都给中等高分"（平均主义型）

错位最大的成员往往是节目里最值得讨论的对象。

**(e) 首推稳定度 (rank dispersion)**：被排序时 rank 的标准差。低 std = 大家对 ta 的位置认知一致；高 std = 评价极化。

**(f) 跨年首推变化**：2025 vs 2026 首推率差值。挑出升幅 / 降幅最大的成员，自然连到 Q23/Q24（最佳进步奖）做交叉验证。

**(g) 首推 demographic profile**：跟 A1 筛选器结合，每位成员的首推者画像 —— 主要来自哪个入坑期？哪个年龄段？女性占比？即每位成员的 "stan 画像卡"，可放在 A4 成员深度页里。

##### 可选

**(h) 集中度指数 (HHI)**：所有成员首推占有率平方求和，得到 0–10000 的单一数字。越高表示首推垄断头部；越低表示百花齐放。一年一个数字做趋势线。

**(i) "0 首推"成员名单**：从未被任何人推为第一的成员清单。残酷但有用，往往是新人 / 即将毕业 / 知名度低的。

#### A7. 成员期别 (Generation) 元数据 + filter

##### 为什么必须加

- 2026 问卷自己就分开问了 Q05（二期/三期）和 Q06（四期生），说明问卷作者**已经承认期别是天然分轴**
- 当前 ranking table 混排有结构性问题：
  - 四期生加入晚，自然下沉，看不出**四期内部**谁出挑
  - 一期生大量毕业，剩下的人占据头部
  - 跨期对比变成"资历竞赛"而非"人气竞赛"
- 加期别后能做：单期内排名、期别内集中度、跨期对比

##### 实现要点

**(a) 新增 `data/members.json` 字典**（项目级元数据，独立于问卷）

```json
{
  "守屋茜": {
    "gen": 1,
    "joinedAt": "2015-08",
    "graduatedAt": "2022-08",
    "status": "graduated",
    "aliases": ["守屋 茜", "守屋アカネ"]
  },
  "山下瞳月": {
    "gen": 4,
    "joinedAt": "2024-12",
    "status": "active",
    "aliases": []
  }
}
```

字段：
- `gen`: 1 / 2 / 3 / 4
- `joinedAt`: 加入日期（便于和入坑期做时间关联）
- `status`: `active` / `graduated`
- `graduatedAt`: 毕业日期（如有）
- `aliases`: 异写名字数组（参考 [js/survey-bi.js:18](js/survey-bi.js:18) `fanEraDefinitions.aliases2026` 的做法）

新一期加入 / 成员毕业时手动更新。

**(b) 在 [scripts/clean_survey_data.py:734](scripts/clean_survey_data.py:734) `aggregate_ranking` 阶段 join**

聚合时把成员名查字典，给每个 item 附加 `gen / status` 字段输出到 `aggregate-*.json`。前端无需自己 join。

字典外的成员（未识别）归入 `gen: "unknown"`，并在 `cleaning-report.json` 中专门列出，方便发现遗漏。

**(c) UI Filter**

成员相关 panel（成员 tab + 涉及 Q03/Q04/Q05/Q06/Q23/Q24/Q25/Q26 的所有 panel）顶部加 chip：
- `全部 / 一期 / 二期 / 三期 / 四期`
- `[ ] 包含毕业生`（默认关闭，理由：当下讨论现役阵容更有意义）

**(d) 期别配色**

给每期分配稳定颜色（建议：一期=深紫、二期=蓝、三期=绿、四期=橙），在跨期对比图里复用，强化视觉认知。

**(e) 与 A6 的协同**

A6 的所有指标都应支持期别 filter：
- "四期生中的首推王"
- "二期生的首推 - Borda 错位最大的人"
- "各期内部的 HHI 集中度对比"

### 二、中价值（值得做，但不紧迫）

#### B1. Cohort × 行为交叉热力图

矩阵热力图：行 = 入坑期，列 = 偶像活动方式（看 MV、买专、抽签…），格子颜色深浅 = 占比。比现有的 `renderMatrixPanel`（实为一排小柱图）信息密度高得多。

#### B2. 单曲 / MV 投票按入坑期分层

"最喜欢的 MV"现在是总榜。但**新粉投新曲、老粉投老曲**是必然 —— 把 Q04/Q09/Q15/Q16 按 cohort 切开，能看到每代粉丝的"代表作"。

#### B3. "时间 / 金钱 / 精力"投入变化的二维分布

[js/survey-bi.js:443-447](js/survey-bi.js:443) 现把 2026-Q09 的三个维度做成三排独立柱图。更有价值的是：
- "时间增加但金钱减少"的人占多少？
- "三项全减少"的人是什么 cohort？

2D/3D 联合分布问题，可用 stacked bar 或 sankey 表达。

#### B4. 痛点 × 影响力的排序（不只是频率）

Q30 购票难点现按出现频率排。**更有意义的指标是"哪个痛点跟没成功购票最相关"**。可做"痛点 → 中签率"散点图：x = 该痛点频率，y = 提及该痛点者的中签率，让真正的瓶颈浮上来。

#### B5. 地理分布

2025-Q37（省市）+ Q38（国家 / 地区）有数据但清洗输出未保留。建议：
- 加一张中国大陆 choropleth（按省份计数）
- 海外用 bar list
- **k-匿名**：单格数 <5 时合并到"其他"，避免小样本可识别

#### B6. 留言精选展示位

未来想开放部分 Q40 / Q38 留言（带受访者同意机制），需要：
- 清洗脚本支持 opt-in 字段（如新增问卷题"是否允许公开您这条留言"）
- 公开 JSON 输出 `samples: []`
- 前端 `renderTextStats` 读取并轮播

把"BI 数据"变成"内容素材"的桥梁，对播客内容生产有直接帮助。

#### B7. 通用 hover tooltip 系统（基础设施）

**优先级前置说明**：虽然分在中价值档（属于基础设施类），但 **A6 上线时强依赖此组件**，所以执行顺序上必须先于 A6。

##### 需求

所有"有口径细节"的字段，鼠标悬停时显示浮动解释窗口。最少需要覆盖：

| 字段 | tooltip 内容（示例） |
|---|---|
| 条件首推率 | 公式 + "在被排名的 N 人里 ta 被推为第一的比例，反映死忠浓度" |
| 绝对首推率 | 公式 + "全样本中把 ta 推为第一的人数占比，反映群众基础" |
| Borda score | 公式 + "第 1 名得 max_rank 分、未排进得 0 分；跨年因候选成员数不同不可直接对比" |
| top1 / top3 / averageRank | 一句话解释 |
| rankedCount | "实际把 ta 排进序列的人数（≤ sampleCount）" |
| skipped_by_logic | "因为分支跳过未答此题的人数，不计入分母" |
| empty_marker | "受访者主动留空但题目要求展示的状态" |
| 95% 置信区间（A2 上线后） | 区间含义 + 计算方法 |
| 显著性标记 ★（A2 上线后） | "该年份对比通过 chi-square 检验 p<0.05" |

##### 实现要点

- 用原生 `<dialog>` 或纯 CSS 浮层即可，不引入第三方库（与 KAGENARE "零依赖"基调一致）
- 触发方式：hover + focus（键盘可访问），移动端长按
- 数据驱动：tooltip 文案统一放在 `js/survey-bi.js` 里一个 `INDICATOR_DEFINITIONS` 对象，避免散落在各 panel
- UI 风格：复用 Zune 灰阶 + 紫色 accent，半透明 + 1px 边框，跟 panel 一致
- ARIA：`aria-describedby` 绑定，屏幕阅读器友好
- **可访问性兜底**：tooltip 内容必须能在某个非悬停状态下也访问到（例如"清理口径"页里列一份完整术语表），不能让信息**只**存在于悬停

##### 与其他条目的关系

- **A6 首推率** 强依赖：两种首推率必须有 tooltip
- **A2 置信区间 / 显著性** 强依赖：统计学概念必须有解释
- **A3 漏斗** 弱依赖：每一步的留存率定义可以用 tooltip 说明
- 长期看是所有指标的"在线词典"，节省 panel 文字描述空间

### 三、低价值（点缀）

### 核心建议

如果只挑两件事做：

1. **A1 筛选器 + 行级脱敏 JSON** —— 把看板从"展示已知结论"升级为"探索未知结论"，是 BI 工具和 infographic 的本质区别。
2. **A5 自由文本词频 / 关键词** —— 中文区粉丝写了几千条文字全被丢掉；用合规方式还原，对"幕前须知"这档**研究型播客**的内容生产是直接生产力。

### 执行顺序建议

A6 + A7 + B7 是一组紧耦合工作，建议作为下一个里程碑：

1. **B7 通用 tooltip 组件** —— 基础设施，先做
2. **A7 成员期别字典 + filter** —— A6 需要按期别切片
3. **A6 首推率 + 衍生指标** —— 上线时两种首推率配 tooltip

这一组完成后，成员 tab 的分析深度会有质的提升。
